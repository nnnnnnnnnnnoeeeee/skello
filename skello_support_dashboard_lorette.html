Architecture du modèle de données

Layer 1 — Staging · parsing JSON, typage, déduplication ETL
STG_CONVERSATIONS Parse ASSIGNEE, CONVERSATION_RATING, TAGS · déduplication sur _SDC_EXTRACTED_AT (QUALIFY ROW_NUMBER)
STG_CONVERSATION_PARTS Parse AUTHOR · exclut les bots (WHERE author_type <> 'bot') · expose part_group propre
STG_CONVERSATION_TAGS LATERAL FLATTEN du JSON TAGS · 1 ligne par tag par conversation
Layer 2 — Marts · agrégations précalculées pour le reporting
FACT_CONVERSATIONS Grain : 1 ligne/conv · FRT (1er msg client → 1ère réponse admin), CSAT, durée, dimensions temporelles
AGG_WEEKLY_TEAM Grain : semaine · CSAT positif %, score moyen, FRT <5min %, FRT médian, conversations prioritaires
AGG_WEEKLY_AGENT Grain : semaine × agent · toutes les métriques ci-dessus par personne (JOIN INNER sur DIM_SUPPORT_AGENTS)
AGG_HOURLY_VOLUME Grain : jour de semaine × heure · heatmap de charge pour le planning
AGG_TAGS_WEEKLY Grain : semaine × tag · thèmes récurrents (LATERAL FLATTEN → agrégation)
AGG_CSAT_DISTRIBUTION Grain : semaine × note · distribution 1–5 avec % — détecte les dérives de qualité
Choix de modélisation clés
  • FRT = écart 1er message client → 1ère réponse admin humain (bots exclus) — mesure la réactivité réelle perçue
  • CSAT positif = note ≥ 4 sur 5 (Intercom : 4 = Bon, 5 = Excellent)
  • Déduplication ETL via QUALIFY ROW_NUMBER OVER (PARTITION BY ID ORDER BY _SDC_EXTRACTED_AT DESC)
  • Dénominateur SLA = conversations ayant reçu une réponse (exclut les convs abandonnées par le client)
  • Conversations non assignées à l'équipe conservées dans FACT mais exclues des marts individuels
